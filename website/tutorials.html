Tutorial 1: Browser Pod with Display & Audio

Run a full GUI browser inside a governed pod — with display forwarding and audio playback.

What You'll Need

  • Google Chrome installed on the host (deb package, not snap)
  • X11 or Wayland display session
  • PulseAudio or PipeWire for audio

Note: Firefox is a snap on Ubuntu 24.04 and doesn't work inside namespace pods. Use Chrome (deb package) instead.

Step 1: Create the Pod

sudo envpod init browser-pod -c examples/browser.yaml

This creates a pod with browser seccomp profile, 256 MB /dev/shm, GPU passthrough, display and audio access, and blacklist DNS.

Step 2: Create a Browser User

Chrome refuses to run as root. Create a non-root user inside the pod:

sudo envpod run browser-pod -- useradd -m browseruser

Step 3: Launch Chrome

sudo envpod run browser-pod -d -a --user browseruser -- google-chrome https://google.com

Flags: -d auto-detects display protocol, -a auto-detects audio, --user browseruser runs as non-root.

Chrome opens inside the pod. All filesystem writes go to the COW overlay, DNS queries are logged, and resource usage is capped.

Step 4: Review What Happened

# See filesystem changes
sudo envpod diff browser-pod

# See network activity (DNS queries)
sudo envpod audit browser-pod

# Accept or discard changes
sudo envpod commit browser-pod     # keep browser profile/cache
# or
sudo envpod rollback browser-pod   # discard everything

Step 5: Check Security

sudo envpod audit --security -c examples/browser.yaml

You'll see findings including I-04 (CRITICAL: X11 keylogging) and I-05 (HIGH: microphone access). See Tutorial 2 to fix these.

Cleanup

sudo envpod destroy browser-pod --full

Tutorial 2: Secure Browser (Wayland + PipeWire)

The same browser pod, but with maximum display and audio security. Drops the display finding from CRITICAL to LOW and audio from HIGH to MEDIUM.

What You'll Need

  • Wayland compositor (GNOME on Wayland, Sway, etc.)
  • PipeWire audio server (default on Ubuntu 23.10+, Fedora 34+)
  • Google Chrome (deb package)

Check your setup:

# Should show "wayland" or a Wayland socket name
echo $WAYLAND_DISPLAY

# Should show PipeWire processes
pw-cli info 0

Step 1: Create the Secure Pod

sudo envpod init browser-secure -c examples/browser-wayland.yaml

Key differences: display_protocol: wayland (no X11 fallback) and audio_protocol: pipewire (no PulseAudio fallback).

Step 2: Create a Browser User

sudo envpod run browser-secure -- useradd -m browseruser

Step 3: Launch Chrome on Wayland

sudo envpod run browser-secure -d -a --user browseruser -- google-chrome --ozone-platform=wayland https://google.com

Step 4: Compare Security

sudo envpod audit --security -c examples/browser-wayland.yaml
Findingbrowser.yamlbrowser-wayland.yaml
I-04 DisplayCRITICAL (X11 keylogging)LOW (Wayland client isolation)
I-05 AudioHIGH (PulseAudio mic access)MEDIUM (PipeWire permissions)

Why Wayland is safer: X11 has no client isolation — any client can capture keystrokes and inject input. Wayland compositors enforce strict client isolation by design.

Why PipeWire is safer: PulseAudio grants unrestricted microphone access. PipeWire provides finer-grained permission controls.

Cleanup

sudo envpod destroy browser-secure --full

Tutorial 3: Headless Browser Automation

Run browser automation (Playwright, Puppeteer, browser-use) inside a pod — no display or audio needed.

Step 1: Create the Pod

sudo envpod init headless -c examples/browser-use.yaml

This installs browser-use and Playwright with Chromium during setup. Uses system_access: advanced for COW overlays on system directories.

Step 2: Run an Automation Script

# Drop into the pod and write a script
sudo envpod run headless -- /bin/bash

Inside the pod:

# Save as /opt/scrape.py
from browser_use import Browser

browser = Browser(headless=True)
page = browser.new_page()
page.goto("https://example.com")
print(page.title())
browser.close()

Or run it directly:

sudo envpod run headless -- python3 /opt/scrape.py

Step 3: Review Network Activity

sudo envpod audit headless

No display or audio findings — headless avoids I-04 and I-05 entirely.

Cleanup

sudo envpod destroy headless --full

Tutorial 4: YouTube in a Sandbox

Watch YouTube (or any streaming video) inside a fully governed pod with display and audio forwarding.

Step 1: Create and Configure

sudo envpod init youtube -c examples/browser-wayland.yaml
sudo envpod run youtube -- useradd -m browseruser

Step 2: Launch Chrome

For Wayland + PipeWire (recommended):

sudo envpod run youtube -d -a --user browseruser -- google-chrome --ozone-platform=wayland https://youtube.com

For X11 + PulseAudio (works everywhere but less secure):

sudo envpod init youtube-x11 -c examples/browser.yaml
sudo envpod run youtube-x11 -- useradd -m browseruser
sudo envpod run youtube-x11 -d -a --user browseruser -- google-chrome https://youtube.com

What's Happening Under the Hood

  1. Display: The host's Wayland socket is bind-mounted into the pod. Chrome renders directly to the host compositor.
  2. Audio: The host's PipeWire socket is bind-mounted. Audio plays through the host's audio stack.
  3. GPU: /dev/nvidia* and /dev/dri/* are bind-mounted. Hardware video decoding with zero virtualization overhead.
  4. Network: DNS blacklist mode logs all domain resolutions. YouTube CDN domains are allowed.
  5. Filesystem: Chrome's profile, cache, and downloads all go to the COW overlay.

Cleanup

sudo envpod destroy youtube --full

Tutorial 5: Legacy Browser (X11 + PulseAudio)

For systems without Wayland or PipeWire — older Ubuntu (20.04, 22.04), Debian, RHEL, or any X11-based desktop.

What You'll Need

  • X11 desktop session (GNOME on X11, XFCE, KDE on X11, i3, etc.)
  • PulseAudio audio server
  • Google Chrome (deb package)
# Should show ":0" or ":1"
echo $DISPLAY

# Should NOT be set (confirms X11, not Wayland)
echo $WAYLAND_DISPLAY

# Should show PulseAudio info
pactl info

Step 1: Create the Pod

sudo envpod init legacy-browser -c examples/browser.yaml

Step 2: Create a Browser User

sudo envpod run legacy-browser -- useradd -m browseruser

Step 3: Launch Chrome

sudo envpod run legacy-browser -d -a --user browseruser -- google-chrome https://google.com

What -d does on X11: Mounts the X11 socket, sets DISPLAY=:0, does NOT set WAYLAND_DISPLAY.

What -a does with PulseAudio: Mounts the PulseAudio socket, copies the cookie, sets PULSE_SERVER.

Security Considerations

X11 + PulseAudio is the least secure display/audio combination:

FindingSeverityWhy
I-04CRITICALX11 has no client isolation — any client can keylog, screenshot, and inject input
I-05HIGHPulseAudio grants unrestricted microphone recording

Acceptable for dedicated dev machines with semi-trusted agents. For production, use Tutorial 2.

Custom X11 + PulseAudio Config

# x11-pulseaudio.yaml
name: legacy-browser
type: standard
backend: native

devices:
  gpu: true
  display: true
  display_protocol: x11          # force X11 (no Wayland fallback)
  audio: true
  audio_protocol: pulseaudio     # force PulseAudio (no PipeWire fallback)

security:
  seccomp_profile: browser
  shm_size: "256MB"

network:
  mode: Isolated
  dns:
    mode: Blacklist
    deny:
      - "*.internal"
      - "*.local"
      - "*.corp"

processor:
  cores: 2.0
  memory: "4GB"
  max_pids: 1024

budget:
  max_duration: "30m"

audit:
  action_log: true

Cleanup

sudo envpod destroy legacy-browser --full

Tutorial 6: GPU ML Training

Run GPU-accelerated machine learning training inside a governed pod.

What You'll Need

  • NVIDIA GPU with drivers installed on host
  • nvidia-smi working on the host

Step 1: Create the Pod

sudo envpod init ml -c examples/ml-training.yaml

GPU passthrough (zero-copy), 4 CPU cores, 16 GB memory, 8-hour max duration, DNS whitelist for PyPI/Hugging Face/PyTorch/Anaconda, auto-installs PyTorch.

Step 2: Verify GPU Access

sudo envpod run ml -- nvidia-smi

GPU passthrough adds ~28ms overhead (namespace entry), after which performance is identical to the host.

Step 3: Run Training

# Interactive
sudo envpod run ml -- /bin/bash

# Or directly
sudo envpod run ml -- python3 -c "
import torch
print(f'CUDA available: {torch.cuda.is_available()}')
print(f'GPU: {torch.cuda.get_device_name(0)}')
x = torch.randn(1000, 1000, device='cuda')
print(f'Tensor on GPU: {x.device}')
"

Step 4: Save Results

# See what the training produced
sudo envpod diff ml

# Commit model checkpoints to host
sudo envpod commit ml /opt/models/
sudo envpod commit ml /opt/results/

# Or export to a specific directory
sudo envpod commit ml --output /tmp/training-output/

# Discard everything else
sudo envpod rollback ml

Tutorial 8: Multi-Agent Fleet

Spin up multiple agent pods from a single base for parallel workloads.

Step 1: Create a Base Pod

sudo envpod base create agent-base -c examples/coding-agent.yaml

Creates a reusable snapshot with all setup commands pre-run.

Step 2: Clone N Agents

for i in $(seq 1 10); do
    sudo envpod clone agent-base "agent-$i"
done

Under a second for 10 clones (~8ms each). Each clone has its own overlay, network namespace, and cgroup. Shared rootfs via symlink.

Step 3: Run Tasks

# Assign each agent a task
sudo envpod run agent-1 -- python3 /opt/task1.py
sudo envpod run agent-2 -- python3 /opt/task2.py
# ...

Step 4: Collect Results

# Review each agent's output
for i in $(seq 1 10); do
    echo "=== agent-$i ==="
    sudo envpod diff "agent-$i"
done

# Export all results to a directory
for i in $(seq 1 10); do
    sudo envpod commit "agent-$i" --output "/tmp/results/agent-$i/"
done

Step 5: Tear Down

sudo envpod destroy agent-1 agent-2 agent-3 agent-4 agent-5 \
    agent-6 agent-7 agent-8 agent-9 agent-10
sudo envpod base destroy agent-base
sudo envpod gc

Tutorial 9: Node.js Development

Set up a full Node.js development environment inside a pod — with nvm, npm, and binaries accessible to both root and non-root users.

The Challenge

Setup commands run as root, but pods default to the non-root agent user (UID 60000). The classic nvm install puts everything under /root/.nvm/ which is mode 700 — the agent user can't access it.

The fix: Install nvm to /opt/nvm (world-accessible) and symlink binaries to /usr/local/bin.

Step 1: Understand the Config

name: nodejs
type: standard
backend: native

filesystem:
  system_access: advanced     # Required for symlinks in /usr/local/bin
  tracking:
    watch:
      - /home
      - /opt
      - /root
      - /workspace
    ignore:
      - /var/cache
      - /var/lib/apt
      - /var/lib/dpkg
      - /tmp
      - /run
      - /opt/nvm             # nvm install dir (large, not project code)
      - /root/.npm           # npm cache
      - /root/.node-gyp      # native addon build cache

network:
  mode: Monitored
  dns:
    mode: Whitelist
    allow:
      - raw.githubusercontent.com
      - "*.githubusercontent.com"
      - nodejs.org
      - "*.nodejs.org"
      - github.com
      - "*.github.com"
      - registry.npmjs.org
      - "*.npmjs.org"
      - "*.npmjs.com"

processor:
  cores: 2.0
  memory: "2GB"

Key decisions: system_access: advanced for writable system dirs, /opt/nvm in tracking ignore to keep envpod diff useful, DNS whitelist for nvm + npm.

Step 2: Create the Pod

sudo envpod init myapp -c examples/nodejs.yaml

During setup, nvm installs to /opt/nvm, Node.js 22 is downloaded, and node/npm/npx are symlinked to /usr/local/bin.

Step 3: Verify Node.js

sudo envpod run myapp -- node -v
sudo envpod run myapp -- npm -v

Step 4: Develop Inside the Pod

# Interactive shell
sudo envpod run myapp -- bash

# Or run commands directly
sudo envpod run myapp -- npm init -y
sudo envpod run myapp -- npm install express
sudo envpod run myapp -- node server.js

Step 5: Install Global npm Packages

sudo envpod run myapp --root -- bash -c '
  export NVM_DIR=/opt/nvm && . "$NVM_DIR/nvm.sh"
  npm install -g typescript
  ln -sf "$(which tsc)" /usr/local/bin/tsc
  ln -sf "$(which tsserver)" /usr/local/bin/tsserver
'

# Now tsc works for all users
sudo envpod run myapp -- tsc --version

Cloning for Multiple Projects

# Clone is ~130ms (vs minutes for full nvm + Node.js install)
sudo envpod clone myapp project-a
sudo envpod clone myapp project-b

# Each clone has its own overlay — independent node_modules
sudo envpod run project-a -- bash -c 'cd /workspace && npm init -y && npm install react'
sudo envpod run project-b -- bash -c 'cd /workspace && npm init -y && npm install express'

Troubleshooting

  • "Cannot change mode: Operation not permitted" — You need system_access: advanced in your config.
  • "node: command not found" — The /usr/local/bin symlinks are missing. Re-run the symlink step.
  • npm install hangs — The npm registry isn't in your DNS whitelist. Add registry.npmjs.org and *.npmjs.org.
  • "EACCES: permission denied" — Run with --root for global installs.

Cleanup

sudo envpod destroy myapp --full

Tutorial 10: Pod-to-Pod Discovery

Let two AI agents find each other by name — without hardcoding IP addresses. One pod exposes a service, the other resolves it as <name>.pods.local.

The Solution

  • allow_discovery: true on the service pod — registers with the central DNS daemon
  • allow_pods: [name] on the client pod — permits lookup
  • envpod dns-daemon — host-side daemon enforcing bilateral policy

Both sides must opt in. If either is missing, *.pods.local → NXDOMAIN.

Step 1: Start the DNS Daemon

sudo envpod dns-daemon

Listens on /var/lib/envpod/dns.sock. Keep running in a dedicated terminal or as a systemd service.

Step 2: Create the Service Pod

sudo envpod init api-service -c examples/discovery-service.yaml

Key config:

network:
  mode: Isolated
  subnet_base: "10.200.50"    # shared subnet
  allow_discovery: true       # register as api-service.pods.local
  internal_ports:
    - "8080"                  # accept connections from other pods

Step 3: Create the Client Pod

sudo envpod init api-client -c examples/discovery-client.yaml
network:
  mode: Isolated
  subnet_base: "10.200.50"    # must match service pod
  allow_pods:
    - api-service             # permitted to resolve api-service.pods.local

Step 4: Run the Service

sudo envpod run api-service -- python3 -m http.server 8080

Step 5: Connect from the Client

sudo envpod run api-client -- curl http://api-service.pods.local:8080/

The daemon verifies both allow_discovery on the target and allow_pods on the client before returning the A record.

Wildcard Discovery

network:
  allow_pods: ["*"]    # resolve all discoverable pods

The * matches only pods with allow_discovery: true — targets must still opt in.

Running as a systemd Service

# /etc/systemd/system/envpod-dns.service
[Unit]
Description=envpod central pod discovery daemon
After=network.target

[Service]
Type=simple
ExecStart=/usr/local/bin/envpod dns-daemon
Restart=on-failure
RestartSec=2

[Install]
WantedBy=multi-user.target
sudo systemctl enable --now envpod-dns

Tear Down

sudo envpod destroy api-service api-client

Tutorial 11: Live Discovery Mutations

Change which pods can see each other — while they are running — without restarting anything.

How It Works

envpod discover sends an UpdateDiscovery message to the daemon over the Unix socket. The daemon updates its registry immediately. The command also writes to pod.yaml for persistence.

Checking Current State

sudo envpod discover api-service
Pod:              api-service
IP:               10.200.50.2
Allow discovery:  yes
Allow pods:       client-pod

Enabling / Disabling Discoverability

# Make a pod discoverable
sudo envpod discover worker-pod --on

# Hide it again (queries return NXDOMAIN immediately)
sudo envpod discover worker-pod --off

Adding / Removing Pods from allow_pods

# Grant a new pod permission to discover
sudo envpod discover api-service --add-pod new-client

# Revoke access
sudo envpod discover api-service --remove-pod new-client

# Revoke all access at once
sudo envpod discover api-service --remove-pod '*'

Combining Flags

# Enable discovery and grant access simultaneously
sudo envpod discover api-service --on --add-pod orchestrator --add-pod monitor-pod

# Disable discovery and clear all in one shot
sudo envpod discover api-service --off --remove-pod '*'

Scenario: Adding a Pod to a Running Fleet

# Grant monitor-pod permission to discover api-service
sudo envpod discover api-service --add-pod monitor-pod

# Make monitor-pod itself discoverable
sudo envpod discover monitor-pod --on --add-pod orchestrator

# Verify
sudo envpod discover api-service
sudo envpod discover monitor-pod

Scenario: Revoking a Compromised Pod

# Remove agent-7 from all allow_pods lists
sudo envpod discover api-service --remove-pod agent-7
sudo envpod discover data-store --remove-pod agent-7

# Turn off agent-7's own discoverability
sudo envpod discover agent-7 --off

agent-7 is now invisible and can't resolve any *.pods.local names. Isolated without being destroyed.

Tutorial 12: Action Catalog — Governed Tool Use

Give an AI agent a menu of allowed actions. The agent discovers what it can do, calls actions by name, and envpod executes them — after validation, tier enforcement, and optional human approval. The agent never makes calls directly.

This is the governed equivalent of MCP tool use.

Where actions.json Lives

The catalog is a plain JSON file in the pod's state directory on the host — outside the overlay, outside the agent's reach.

/var/lib/envpod/pods/<pod-name>/
  actions.json        ← the action catalog  (host-only)
  pod.yaml            ← pod config
  audit.jsonl         ← audit log
  upper/              ← agent's COW overlay
  merged/             ← agent's filesystem view
  rootfs/             ← base snapshot

3 Ways to Add Actions

Option 1 — CLI (fastest for one-offs):

sudo envpod actions my-agent add fetch_api \
  --description "Fetch data from the upstream API" \
  --type http_get \
  --tier immediate

Option 2 — Edit actions.json directly:

sudo nano /var/lib/envpod/pods/my-agent/actions.json

Changes are picked up immediately. No pod restart needed.

Option 3 — pod.yaml setup commands (reproducible):

# pod.yaml
setup:
  - cp /home/mymark/configs/my-agent-actions.json \
       /var/lib/envpod/pods/my-agent/actions.json

1. Create the Pod

# pod.yaml
name: my-agent
network:
  mode: Filtered
  allow:
    - api.github.com
    - github.com
    - hooks.slack.com
    - pypi.org
queue:
  socket: true
sudo envpod init my-agent -c pod.yaml

2. Create the Action Catalog

sudo tee /var/lib/envpod/pods/my-agent/actions.json > /dev/null << 'EOF'
[
  {
    "name": "fetch_data",
    "description": "Fetch data from the upstream API",
    "action_type": "http_get",
    "tier": "immediate",
    "params": [

      {"name": "headers", "required": false}
    ]

3. Store Credentials

echo "ghp_xxxxxxxxxxxx" | sudo envpod vault my-agent set GITHUB_TOKEN

Encrypted at rest. The agent never sees it — envpod fetches it from the vault at execution time.

4. Write the Agent

# agent.py
import socket, json, time

def send(msg):
    """Send a request to the envpod queue socket and return the response."""
    sock = socket.socket(socket.AF_UNIX, socket.SOCK_STREAM)
    sock.connect("/run/envpod/queue.sock")
    sock.sendall((json.dumps(msg) + "\n").encode())
    data = b""
    while True:
        chunk = sock.recv(4096)
        if not chunk:
            break
        data += chunk
        if b"\n" in data:
            break
    sock.close()
    return json.loads(data.strip())

# Step 1: Discover the tool menu
actions = send({"type": "list_actions"})["actions"]
print("Available actions:")
for a in actions:
    print(f"  {a['name']:20} tier={a['tier']:10} type={a.get('action_type','custom')}")

# Step 2: Fetch data (immediate — executes without approval)
print("\nFetching data...")
r = send({
    "type": "call",
    "action": "fetch_data",
    "params": {"url": "https://api.github.com/repos/torvalds/linux/commits?per_page=5"}
})
print(f"fetch_data: {r['status']}  id={r['id']}")

# Step 3: Save results (immediate)
result = {"fetched_at": "2026-03-03T14:22:00Z", "count": 5}
r = send({
    "type": "call",
    "action": "save_output",
    "params": {
        "path": "/workspace/output.json",
        "content": json.dumps(result, indent=2)
    }
})
print(f"save_output: {r['status']}  id={r['id']}")

# Step 4: Commit (staged — waits for human approval)
r = send({
    "type": "call",
    "action": "commit_work",
    "params": {"message": "feat: add fetched commit data"}
})
print(f"commit_work queued: {r['id']} (status: {r['status']})")
commit_id = r["id"]

# Step 5: Notify via webhook (staged)
r = send({
    "type": "call",
    "action": "notify_complete",
    "params": {
        "url": "https://hooks.slack.com/services/T.../B.../xxx",
        "payload": json.dumps({"text": f"Agent complete. {result['count']} records saved."})
    }
})
print(f"notify_complete queued: {r['id']} (status: {r['status']})")
notify_id = r["id"]

# Step 6: Poll for approvals
print("\nWaiting for host approval...")
for action_id in [commit_id, notify_id]:
    while True:
        status = send({"type": "poll", "id": action_id})["status"]
        if status in ("executed", "cancelled", "blocked"):
            print(f"  {action_id}: {status}")
            break
        time.sleep(2)

print("\nDone.")

5. Run the Pod

sudo envpod run my-agent -- python3 agent.py

fetch_data and save_output execute immediately. commit_work and notify_complete are staged — the agent blocks on the poll loop.

6. Review and Approve from the Host

# See staged actions waiting for approval
sudo envpod queue my-agent

# Inspect full params before approving
sudo envpod queue my-agent --id a1b2c3 --json

# Approve
sudo envpod approve my-agent a1b2c3
sudo envpod approve my-agent a1b2c4

# Or cancel
sudo envpod cancel my-agent a1b2c4

envpod fetches GITHUB_TOKEN from the vault and runs the git commit. The agent never saw the token.

sudo envpod audit my-agent

Hot-Reload the Catalog

# Decide mid-run that push should require approval
sudo envpod actions my-agent set-tier push_work staged

# Block the webhook entirely for this run
sudo envpod actions my-agent set-tier notify_complete blocked

The catalog is re-read on every list_actions call. No restart needed.

Tips & Troubleshooting

Chrome Won't Start

  • Chrome refuses to run as root. Always use --user browseruser.
  • On Wayland, Chrome needs --ozone-platform=wayland.
  • Firefox is a snap on Ubuntu 24.04 — snaps don't work in namespace pods. Use Chrome.

No Audio

  • Check pw-cli info 0 (PipeWire) or pactl info (PulseAudio) on the host.
  • Verify the socket is mounted: sudo envpod run pod -a -- ls -la /tmp/pipewire-0
  • PulseAudio requires cookie auth. envpod auto-copies it, but check permissions if audio fails.

GPU Not Detected

  • Verify nvidia-smi works on the host first.
  • Check devices.gpu: true in your pod config.
  • If /dev/nvidia* and /dev/dri/* don't exist on the host, there's nothing to mount.

DNS Queries Failing

  • Check sudo envpod audit pod-name for DNS deny/allow decisions.
  • Use sudo envpod dns pod-name --allow domain.com to add a domain live.
  • Whitelist mode: only allowed domains resolve. Blacklist mode: everything except denied domains.

Pod Discovery Not Working

  • Verify envpod dns-daemon is running: ls -la /var/lib/envpod/dns.sock
  • Service pod needs allow_discovery: true and must start after the daemon.
  • Client pod needs allow_pods: ["service-name"] (or ["*"]).
  • Both pods must use the same subnet_base.
  • Check sudo envpod audit client-pod | grep pods.local — dns_deny = policy blocked; no entry = daemon unreachable.

Pod Creation Slow

  • envpod init takes ~1.3s (builds rootfs). Use envpod clone for subsequent pods (~130ms).
  • First envpod run after init takes ~1.3s (cold start). Subsequent runs: 20-45ms.
